## Where We Left Off

Stage 8 solved a privacy problem.

The system could already register a file hash on-chain and later prove the file
existed at that point in time. Stage 7 extended this by storing a copy of the
file on IPFS — useful for retrieval, but it exposed the file's content to
anyone who knew the CID. Stage 8 addressed that: encrypt the file locally
**before** uploading to IPFS, so the public network only ever stores
ciphertext. The original file's SHA-256 is still what gets registered on-chain.
The password never leaves the owner's hands.

```
Stage 8:
file → AES-256-GCM encrypt → IPFS (ciphertext only)
                           → register(SHA-256(original))
```

Privacy: solved. The on-chain record still anchors the original plaintext, but
the file itself is never exposed.

So what problem is left?

**Scale.**

## The Gas Cost Problem

Imagine registering fifty works — poems, essays, designs. Each one needs a
separate call to `register(bytes32 fileHash, string uri)`. That is fifty
transactions, fifty gas fees, fifty confirmations to wait for.

The cost is linear in the number of files. For a small portfolio this is
manageable. For a batch of hundreds of works published together — a music
album, a design collection, a set of academic papers — it becomes impractical.

```
file A → register(hashA)   transaction 1
file B → register(hashB)   transaction 2
file C → register(hashC)   transaction 3
...
file N → register(hashN)   transaction N
```

Stage 9 asks: can we prove the whole batch with **one** transaction?

---

## The Merkle Tree Answer

A Merkle tree is a binary hash tree. Each leaf is the hash of one piece of
data. Each internal node is the hash of its two children concatenated. The
root of the tree is a single hash that commits to every leaf.

```
leaf_0 = SHA-256(file_A)
leaf_1 = SHA-256(file_B)
leaf_2 = SHA-256(file_C)
leaf_3 = SHA-256(file_C)   ← duplicate, because 3 is odd

parent_0 = SHA-256(leaf_0 || leaf_1)
parent_1 = SHA-256(leaf_2 || leaf_3)

root = SHA-256(parent_0 || parent_1)
```

The root is fixed-length — 32 bytes, the same as any other SHA-256 hash. The
existing smart contract's `register(bytes32, string)` accepts it without
modification.

```
Stage 9:
file A hash ──┐
file B hash ──┼── Merkle tree ── root ── register(root)   one transaction
file C hash ──┘
```

The blockchain now carries the root. The root commits to every file hash. One
transaction proves the entire batch.

---

## What a Merkle Proof Is

Registering the root is only half the story. The other half is proving that a
specific file belonged to the batch.

A Merkle proof is the minimal set of sibling hashes a verifier needs to
recompute the root starting from just one leaf. For a file at position 1 in a
four-leaf tree, the proof is two items:

```
Step 1: sibling of leaf_1 is leaf_0 (position: left)
        current = SHA-256(leaf_0 || leaf_1) = parent_0

Step 2: sibling of parent_0 is parent_1 (position: right)
        current = SHA-256(parent_0 || parent_1) = root
```

If the final `current` equals the registered root, the file was in the batch.
No other files need to be known. No trusted party needs to confirm it.

```
file + proof → recompute root → compare with on-chain root → verified
```

This is the core insight: **the blockchain proves the batch; the Merkle proof
proves one file belongs to that batch**.

---

## Determinism Is Non-Negotiable

A Merkle root depends on the order of its leaves. If two parties build the
tree with the same files but in different orders, they get different roots.
That would mean a proof generated by the registration tool fails verification
by an independent third party.

Stage 9 fixes the leaf order to a single canonical rule:

> Leaves are sorted by their **normalised relative file path**, ascending
> lexicographic order.

Normalisation means: replace backslashes with forward slashes, remove any
leading `./`. The result is the same on Windows, macOS, and Linux.

```python
# sort key
def sort_key(rel: str | Path) -> str:
    return normalize_relative_path(rel).lower()

files = sorted(collected, key=lambda p: sort_key(p.relative_to(folder)))
```

All Merkle tree rules — leaf hash algorithm, internal node algorithm, leaf
ordering, odd-leaf duplication strategy — are recorded in `batch_evidence.json`
alongside the root and transaction hash. A verifier who has never seen this
codebase can read those rules and reproduce the root independently.

## A Bug Worth Documenting

During implementation, a subtle Python pitfall corrupted every hash the code
produced. The original `normalize_hash` function was written as:

```python
h = hex_hash.lower().lstrip("0x")   # ← wrong
```

This looks like "remove the `0x` prefix." It is not. `str.lstrip(chars)` strips
all leading characters that appear anywhere in `chars` — treating it as a
**character set**, not a prefix string.

For the hash `0x0abc1234...`, `lstrip("0x")` strips `'0'`, then `'x'`, then the
next `'0'` — because `'0'` is still in the strip set — leaving `abc1234...`: 63
characters instead of 64. Prepending `"0x"` back gives `"0xabc1234..."`, a
completely different 256-bit value.

In a test suite, this only surfaces when a hash happens to begin with `0x0` —
which occurs with roughly 1/16 probability per hash. The first test with four
leaves exposed it immediately:

```
ValueError: Expected 64-character hex hash, got 63:
    '0x084fed08b978af4d7d196a7446a86b58009e636b611db16211b65a9aadff29c5'
```

The fix is one character shorter and unambiguous:

```python
# Wrong — strips character set {'0', 'x'}:
h = hex_hash.lower().lstrip("0x")

# Correct — removes exactly the two-character prefix:
s = hex_hash.lower()
h = s[2:] if s.startswith("0x") else s

# Or, on Python 3.9+:
h = hex_hash.lower().removeprefix("0x")
```

The general rule: use `lstrip` / `rstrip` / `strip` to strip **characters**;
use slicing or `removeprefix` / `removesuffix` to strip **substrings**.

## What the Smart Contract Sees

The contract was not changed. It still exposes:

```solidity
function register(bytes32 fileHash, string calldata uri) external
function verify(bytes32 fileHash) external view returns (address, uint256, string)
```

The Merkle root is a valid `bytes32`. From the contract's perspective, it is
indistinguishable from any other 32-byte value. The distinction lives entirely
in `batch_evidence.json`:

```json
{
  "record_type": "merkle_batch",
  "merkle_root": "0x9c1b...",
  "uri": "batch://batch_20260627_143012",
  ...
}
```

`record_type = "merkle_batch"` is the signal that the registered value is a
Merkle root, not a single file hash. Any code or verifier that reads the
evidence JSON understands the difference. The on-chain record is simply a
commitment; the evidence package carries the meaning.

## What This Proves — and What It Does Not

If a verifier recomputes the root from a file and its proof, and the computed
root matches the value registered on-chain, they have confirmed:

1. The file's SHA-256 was included in the batch before the blockchain
   transaction was mined.
2. The registration was made by the address shown as `owner`.
3. The proof file was generated from the same tree as the registered root.

What this does **not** prove:

- Legal authorship or copyright ownership.
- That the registrant created the work, not merely that they registered it first.
- Anything that requires a notary, a court, or a jurisdiction.

A Merkle proof is a cryptographic membership certificate, not a legal one.
